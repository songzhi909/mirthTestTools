# Mirth 消息追踪工具 - 功能需求文档

## 项目概述

为 HIS、检查系统、PACS 等医疗信息系统间的消息推送提供可视化追踪与调试能力，替代手动 SQL 查询，提升排障效率。

---

## FR-001 选择消息类型

**优先级**: P0

用户首先选择消息类型（如检查申请、检验申请、处置申请），系统根据该类型加载对应的 XML 字段定义和流程配置。

| 项目 | 说明 |
|------|------|
| 输入 | 消息类型下拉选择 |
| 输出 | 动态加载查询条件面板和流程面板 |
| 配置 | `message_types.json` 中定义每种类型对应的 XML 字段和通道流程 |

---

## FR-002 动态查询条件

**优先级**: P0

根据所选消息类型的 XML 结构，动态展示查询条件输入项。不同消息类型对应不同的业务字段。

**示例 - 检查申请新增（TJ001）：**

| 查询条件 | 来源 | 说明 |
|----------|------|------|
| EXAM_NO | `<EXAM_NO>20058752</EXAM_NO>` | 检查申请号，必填 |
| 时间范围 | Mirth 消息接收时间 | 下拉选择：最近1h/6h/24h/3天/7天 |

**示例 - 检验申请（JY001）：**

| 查询条件 | 来源 | 说明 |
|----------|------|------|
| ORDER_NO | `<ORDER_NO>xxx</ORDER_NO>` | 检验申请号，必填 |
| 时间范围 | Mirth 消息接收时间 | 下拉选择 |

**配置方式**：在 `message_types.json` 中为每种类型定义 `query_fields`，指定 XML 标签名和显示名称。

---

## FR-003 流程面板展示

**优先级**: P0

查询后按业务流程顺序展示各环节的输入/输出面板。每个流程节点显示为一个可点击的面板卡片。

**布局结构（以检查申请新增为例）：**

```
┌─────────────────────────────────────────────────────────────────┐
│  消息类型: [检查申请新增 ▼]   EXAM_NO: [________]  时间: [24h ▼]  [查询] │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐       │
│  │  HIS消息推送  │───→│ TJ001_V3生成 │───→│  V3推送PACS  │       │
│  │              │    │              │    │              │       │
│  │  输入 ✓      │    │  输入 ✓      │    │  输入 ✓      │       │
│  │  输出 ✓      │    │  输出 ✓      │    │  输出 ✓      │       │
│  │  状态: 成功   │    │  状态: 成功   │    │  状态: 失败   │       │
│  └──────────────┘    └──────────────┘    └──────────────┘       │
│                                                                 │
├─────────────────────────────────────────────────────────────────┤
│  [详情面板区域 - 点击上方面板后显示]                               │
└─────────────────────────────────────────────────────────────────┘
```

**每个流程面板显示：**
- 通道名称
- 输入状态（有/无数据，用图标或颜色标识）
- 输出状态（有/无数据）
- 整体状态（成功/失败/无数据）

**流程节点配置**：在 `message_types.json` 的 `channels` 数组中按顺序定义，系统按数组顺序渲染流程图。

## FR-004 面板详情查看

**优先级**: P0

点击任意流程面板，在下方详情区域展示该环节的完整信息，包含多个 Tab 页。

| Tab 页 | 内容 | 数据来源 |
|--------|------|----------|
| 输入XML | 上游发送到本环节的原始报文 | `content_type = 1` |
| 输出XML | 本环节发送给下游的报文 | `content_type = 1`（下一环节的输入）或响应 |
| 变量 | Mirth Channel 变量 | `content_type = 10` |
| 响应 | 下游系统返回的响应 | `content_type = 10/11` 中的响应变量 |
| 错误 | 错误堆栈信息 | `content_type = 9` |

**详情区域布局：**

```
┌─────────────────────────────────────────────────────────────────┐
│  [输入XML] [输出XML] [变量] [响应] [错误]                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  <exam_info>                                                    │
│    <EXAM_NO>20058752</EXAM_NO>                                  │
│    <PATIENT_ID>P001</PATIENT_ID>                                │
│    ...                                                          │
│  </exam_info>                                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## FR-005 消息状态标识

**优先级**: P1

流程面板通过颜色或图标直观展示状态：

| 状态 | 标识 | 含义 |
|------|------|------|
| 成功 | 绿色 / ✓ | 消息已成功发送并被接收 |
| 错误 | 红色 / ✗ | 处理失败，可点击查看错误详情 |
| 无数据 | 灰色 / - | 该环节未查到消息记录 |
| 进行中 | 黄色 / ⟳ | 已接收但未完成处理 |

## FR-006 消息类型配置化

**优先级**: P0

支持通过配置文件扩展新的消息类型，无需修改代码。

**配置文件**: `message_types.json`

**配置结构：**

```json
{
  "消息类型名称": {
    "code": "业务代码",
    "query_fields": [
      {
        "label": "显示名称",
        "xml_tag": "XML标签名",
        "required": true
      }
    ],
    "channels": [
      {
        "name": "环节名称",
        "id": "Channel ID",
        "table_suffix": "表后缀",
        "response_var": "响应变量名",
        "response_content_type": 10
      }
    ]
  }
}
```

**字段说明：**

| 字段 | 必填 | 说明 |
|------|------|------|
| `code` | 是 | 业务代码（如 TJ001） |
| `query_fields` | 是 | 查询条件列表，从 XML 中提取 |
| `query_fields[].label` | 是 | 界面显示名称 |
| `query_fields[].xml_tag` | 是 | XML 中的标签名，用于 LIKE 查询 |
| `query_fields[].required` | 否 | 是否必填，默认 false |
| `channels` | 是 | 流程节点数组，按业务顺序排列 |
| `channels[].name` | 是 | 环节显示名称 |
| `channels[].id` | 是 | Mirth Channel ID |
| `channels[].table_suffix` | 是 | 数据库表后缀（通常与 id 相同） |
| `channels[].response_var` | 否 | 响应报文中的变量名 |
| `channels[].response_content_type` | 否 | 响应 content_type（10 或 11） |

## FR-007 数据库连接配置

**优先级**: P0

支持通过配置文件修改数据库连接信息，无需重新打包。

**配置文件**: `config.ini`

```ini
[database]
host = 127.0.0.1
port = 3306
user = root
password = xxx
database = mirth_data
```

---

## FR-008 异步查询

**优先级**: P1

数据库查询在后台线程执行，不阻塞 GUI 界面。查询过程中显示加载状态。
